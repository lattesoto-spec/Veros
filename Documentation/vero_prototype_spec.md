# Vero Prototype: Build Spec

Last updated: May 2026

This document describes everything needed to build and deploy the Vero prototype. The goal is a working web app you can put in front of a facility manager and ask: "If you could see this every morning, would it change how you staff?"

Not an MVP. A prototype. One user, one facility, CSV uploads, no integrations.

---

## Stack

- Python 3.12, Flask, SQLite, Jinja2 templates
- Chart.js via CDN (one bar chart, no npm)
- Deployed on Fly.io free tier
- No frontend framework, no React, no build step

## CSV upload formats

The user uploads two CSV files. These are the expected columns.

### residents.csv

```
resident_id,name,ancc_class,admitted_date,discharged_date
R001,Joan Smith,13-04,2024-03-15,
R002,Peter Jones,08-02,2023-11-01,
R003,Mary Brown,13-04,2024-06-20,2026-04-10
```

- `discharged_date` is blank for current residents
- `ancc_class` is the AN-ACC classification code. For the prototype, this is stored but not used in calculation. The facility's overall target (`ancc_target` on the facility record) is what matters. In a real product, individual classifications would feed into a weighted target calculation.

### shifts.csv

```
staff_id,staff_name,role,date,start_time,end_time,is_direct_care
S001,Jane Doe,RN,2026-05-01,07:00,15:00,true
S002,Tom Lee,PCA,2026-05-01,07:00,15:00,true
S003,Sam Park,EN,2026-05-01,15:00,23:00,true
S001,Jane Doe,RN,2026-05-01,15:00,19:00,false
```

- `role` must be one of: RN, EN, PCA
- `is_direct_care` = false means the shift was administrative, training, etc. and does not count

On upload:
1. Parse CSV
2. Upsert staff records (match on `staff_id`)
3. Upsert resident records (match on `resident_id`)
4. Insert shift records
5. Redirect to dashboard

Validation: if required columns are missing, show an error on the upload page. No silent failures.

## Care minutes calculation (care_minutes.py)

This is the only business logic in the prototype.

### Per day

```
active_residents = count of residents where admitted_date <= date AND (discharged_date IS NULL OR discharged_date > date)

total_direct_care_minutes = sum of (end_time - start_time) in minutes for all shifts on that date where is_direct_care = true

rn_direct_care_minutes = same filter but only role = 'RN'

care_minutes_per_resident = total_direct_care_minutes / active_residents
rn_minutes_per_resident = rn_direct_care_minutes / active_residents
```

### Per period (week, month, quarter)

Average the daily `care_minutes_per_resident` and `rn_minutes_per_resident` across all days in the period that have shift data.

### Status logic

Compare against facility targets:

```
gap = care_minutes_per_resident - facility.ancc_target
rn_gap = rn_minutes_per_resident - facility.rn_target

if gap >= 0: status = "on_track"        # green
if -5% <= gap < 0: status = "at_risk"   # amber
if gap < -5%: status = "behind"         # red
```

Same logic applies separately for the RN sub-target.

The 5% threshold matters because the Commission triggers a regulatory response when a facility misses by more than 5% for two consecutive quarters.

## Routes (app.py)

### GET /

Redirect to /dashboard if data exists. Otherwise redirect to /upload.

### GET /upload

Render the upload page. Two file inputs: one for residents CSV, one for shifts CSV. A text field for facility name, and number fields for the AN-ACC target and RN target (pre-filled with 215 and 44).

### POST /upload

Process both CSVs. Create or update the facility, residents, staff, and shifts. Redirect to /dashboard.

### GET /dashboard

The main screen. Shows:

1. **Header**: Facility name, today's date, resident count
2. **Today's snapshot**: Total care minutes per resident today vs target. RN minutes per resident today vs target. Two big numbers with green/amber/red background.
3. **Trend chart**: Bar chart (Chart.js) showing daily care minutes per resident for the last 14 days. Horizontal line at the target. Bars colored green/amber/red based on status.
4. **Quarter-to-date average**: Running average for the current quarter. This is what gets reported. Show how far above or below target.
5. **Summary table**: Last 14 days, one row per day. Columns: date, active residents, total care mins, RN care mins, care mins/resident, RN mins/resident, status.

### GET /facility

Facility detail. Shows resident list with AN-ACC class, staff list with roles, and allows updating the facility targets.

### POST /facility/targets

Update the facility's `ancc_target` and `rn_target`. Redirect to /dashboard.

### POST /clear

Delete all data. Start fresh. Useful during demos.

That is 6 routes. Nothing else.

## Templates

### base.html

- Simple HTML5 layout
- Nav bar with two links: Dashboard, Upload
- Chart.js loaded from CDN: `<script src="https://cdn.jsdelivr.net/npm/chart.js"></script>`
- Link to style.css
- No JavaScript framework. No build tools.

### dashboard.html

- Two "big number" cards at the top (total care mins/resident, RN care mins/resident)
- Background color of each card set by status (green/amber/red via CSS class)
- One Chart.js bar chart rendered inline with a `<canvas>` tag and a `<script>` block
- Chart data passed from Flask via `{{ chart_data | tojson }}`
- Summary table below the chart
- "Upload new data" link at the bottom

### upload.html

- A form with:
  - Facility name (text input)
  - AN-ACC target (number input, default 215)
  - RN target (number input, default 44)
  - Residents CSV (file input)
  - Shifts CSV (file input)
  - Submit button
- If validation fails, show error messages above the form

### facility.html

- Resident table: ID, name, AN-ACC class, admitted, discharged
- Staff table: ID, name, role
- Form to update facility targets

## Sample data (seed.py)

A script that generates realistic demo data and writes it to `sample_data/residents.csv` and `sample_data/shifts.csv`.

Generate:
- 1 facility: "Sunrise Aged Care" with target 217 min and RN target 44 min
- 85 residents (realistic for a mid-size facility), 5 discharged in the last quarter
- 30 staff: 6 RNs, 6 ENs, 18 PCAs
- 30 days of shift data with realistic patterns:
  - Morning shift (07:00-15:00): 2 RNs, 2 ENs, 6 PCAs
  - Afternoon shift (15:00-23:00): 1 RN, 2 ENs, 5 PCAs
  - Night shift (23:00-07:00): 1 RN, 1 EN, 3 PCAs
  - Vary staffing slightly day to day (random +/- 1 staff per role per shift)
  - Mark ~90% of shifts as direct care
  - Build in a realistic pattern: some days hit target, some miss, overall slightly below target so the dashboard shows amber/red and creates urgency

The sample data should tell a story: "You are currently averaging 210 minutes against a 217-minute target. You are 3.2% below. One more quarter like this and the Commission will be in contact."

## Deployment (Fly.io)

### Dockerfile

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8080
CMD ["gunicorn", "--bind", "0.0.0.0:8080", "app:app"]
```

### fly.toml (generated by `fly launch`, then edit)

Key settings:
- `internal_port = 8080`
- Mount a persistent volume at `/data` for the SQLite file
- Set env var `DATABASE_PATH=/data/vero.db`

### Deploy commands

```bash
fly launch                                    # creates app
fly volumes create vero_data --size 1         # 1GB persistent volume
# edit fly.toml to add the volume mount
fly deploy                                    # live at vero-xxx.fly.dev
```

### requirements.txt

```
flask==3.1.1
gunicorn==23.0.0
flask-sqlalchemy==3.1.1
```

Three dependencies. That is it.

## What this prototype does NOT include

All of these are MVP scope, not prototype scope:

- User authentication or multi-tenancy
- HumanForce or iCare API integration
- SIRS incident tracking
- PDF report generation
- Email alerts or notifications
- Multi-facility views
- Role-based access control
- Data export
- AN-ACC weighted target calculation per resident
- Historical quarter comparisons
- Mobile-responsive design (nice to have but not blocking)

## Definition of done

The prototype is done when:

1. You can visit the Fly.io URL in a browser
2. Upload two CSVs (or use sample data)
3. See a dashboard showing today's care minutes per resident vs target
4. See a 14-day trend chart with green/amber/red bars
5. See the quarter-to-date average and the gap
6. A facility manager can look at it and immediately understand whether they are on track

That is the entire scope. Build this, put it in front of someone, and see if they care.
